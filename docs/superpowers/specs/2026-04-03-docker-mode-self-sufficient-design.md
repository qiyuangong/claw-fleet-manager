# Docker Mode — Self-Sufficient Fleet Management

**Date:** 2026-04-03  
**Status:** Approved

## Goal

Make Docker mode in claw-fleet-manager self-sufficient, eliminating the dependency on `../agent-fleet/openclaw/scripts/setup.sh` for initial fleet setup. Users should be able to configure, initialize, and manage a full openclaw Docker fleet entirely through the fleet manager.

## Background

Currently, Docker mode assumes configs already exist. The `agent-fleet` repo's `setup.sh` is responsible for:
- Writing per-instance `openclaw.json` (model provider, gateway auth, tools policy, agents config)
- Setting `OPENCLAW_IMAGE` in `fleet.env`

Without running `setup.sh` first, new Docker instances start with no config and the gateway fails to start. The fleet manager cannot be used standalone for Docker mode.

## Scope

Four independent areas of change:

1. **Full instance config generation** — write `openclaw.json` on instance creation
2. **FleetConfig additions** — `OPENCLAW_IMAGE` and `ENABLE_NPM_PACKAGES`
3. **Plugin support in Docker mode** — routes + UI ungate + optional npm cache mount
4. **Fleet Config UI additions** — OPENCLAW_IMAGE, API key input, npm packages toggle

## 1. Instance Config Generation

### What changes

`ComposeGenerator.generate()` writes a full `openclaw.json` for each new instance directory (skipped if the file already exists, preserving user customizations).

### Config shape

```json
{
  "models": {
    "mode": "merge",
    "providers": {
      "default": {
        "baseUrl": "<BASE_URL>",
        "apiKey": "<API_KEY>",
        "api": "openai-completions",
        "models": [{
          "id": "<MODEL_ID>",
          "name": "<MODEL_ID>",
          "api": "openai-completions",
          "reasoning": false,
          "input": ["text"],
          "cost": { "input": 0, "output": 0, "cacheRead": 0, "cacheWrite": 0 },
          "contextWindow": 200000,
          "maxTokens": 65536
        }]
      }
    }
  },
  "agents": {
    "defaults": {
      "compaction": { "mode": "safeguard" },
      "sandbox": { "mode": "non-main", "scope": "session" }
    },
    "list": [{ "id": "main" }]
  },
  "commands": {
    "native": "auto",
    "nativeSkills": "auto",
    "restart": true,
    "ownerDisplay": "raw"
  },
  "gateway": {
    "mode": "local",
    "auth": { "mode": "token", "token": "<TOKEN_i>" },
    "controlUi": {
      "allowedOrigins": [
        "http://127.0.0.1:<GW_PORT>",
        "http://localhost:<GW_PORT>"
      ]
    }
  },
  "tools": {
    "exec": { "security": "deny", "ask": "always" },
    "fs": { "workspaceOnly": true }
  },
  "plugins": { "allow": [] }
}
```

### Value sources

| Field | Source |
|-------|--------|
| `BASE_URL` | `fleet.env` raw var `BASE_URL` |
| `API_KEY` | `fleet.env` raw var `API_KEY` (unmasked via `readFleetEnvRaw()`) |
| `MODEL_ID` | `fleet.env` raw var `MODEL_ID` |
| `TOKEN_i` | Per-instance token from `.env` (already generated by `ComposeGenerator`) |
| `GW_PORT` | `BASE_GW_PORT + (i - 1) * portStep` |

### Tailscale additions

When Tailscale is configured, the generated config additionally sets:
- `gateway.auth.allowTailscale: true`
- `gateway.controlUi.allowInsecureAuth: true`
- `allowedOrigins: ["https://<hostname>:<tsPort>"]`

### Skip condition

If `openclaw.json` already exists for an instance, generation is skipped. This preserves user-edited configs across `docker compose up` regenerations.

### Where BASE_URL / MODEL_ID are blank

If `BASE_URL` or `MODEL_ID` are not set in `fleet.env`, the generated config omits the `models.providers.default` block entirely, so openclaw starts without a configured provider. The user can then configure it via the Config tab per-instance.

## 2. FleetConfig Additions

### New fields in `FleetConfig`

```ts
openclawImage: string;      // OPENCLAW_IMAGE, default 'openclaw:local'
enableNpmPackages: boolean; // ENABLE_NPM_PACKAGES, default false
```

### `FleetConfigService.readFleetConfig()` additions

```ts
openclawImage: vars.OPENCLAW_IMAGE ?? 'openclaw:local',
enableNpmPackages: vars.ENABLE_NPM_PACKAGES === 'true',
```

### Compose template

`ComposeGenerator` replaces the docker-compose variable substitution `${OPENCLAW_IMAGE:-openclaw:local}` with a literal value read from `readFleetEnvRaw()['OPENCLAW_IMAGE'] ?? 'openclaw:local'`, written directly into the compose yaml — consistent with how `CPU_LIMIT`, `MEM_LIMIT`, etc. are handled.

### `.npm` bind mount

When `ENABLE_NPM_PACKAGES=true`, each service in the generated compose gets an additional volume:

```yaml
- ${configDir}/.npm:/home/node/.npm
```

This is a per-instance writable npm cache on the host. npm's default cache path (`~/.npm` = `/home/node/.npm`) is in the read-only container layer by default; this mount makes it writable without tmpfs or env overrides.

The mount is independent of whether users use the plugin UI — it enables any npm operation inside the container (e.g. running `docker exec openclaw-1 node dist/index.js plugins install <spec>` directly).

## 3. Plugin Support in Docker Mode

### Server: route registration

Plugin management routes are split out of the profiles-only block in `index.ts`:

**Stays profiles-only:**
- `GET /api/fleet/profiles`
- `POST /api/fleet/profiles`
- `DELETE /api/fleet/profiles/:name`

**Registers for both modes:**
- `GET /api/fleet/:id/plugins`
- `POST /api/fleet/:id/plugins/install`
- `DELETE /api/fleet/:id/plugins/:pluginId`

These routes call `app.backend.execInstanceCommand(id, ['plugins', ...])`. In Docker mode, `DockerBackend.execInstanceCommand` already runs `docker exec <id> node dist/index.js <args>`, so no backend changes are needed.

### Why plugin install works end-to-end

`openclaw plugins install <spec>` inside a Docker container:
1. Downloads the package via npm → writes to `/home/node/.npm` (writable if `ENABLE_NPM_PACKAGES=true`)
2. Installs plugin files to `/home/node/.openclaw/extensions/<id>/` (config bind mount — always writable, persists across restarts)
3. Enables the plugin and adds it to `plugins.allow` in `/home/node/.openclaw/openclaw.json` (config bind mount — always writable, persists)
4. Outputs "Restart the gateway to load plugins" — user restarts via the instance panel

### Web: PluginsTab

Remove the `instance.profile` guard:

```ts
// Before:
if (!instance.profile) return <section>profile mode only</section>;

// After: removed. Tab renders for Docker instances too.
```

Change query `enabled` condition:
```ts
// Before:
enabled: Boolean(instance.profile),

// After:
enabled: true,
```

If npm packages are not enabled and a user tries to install from npm, the error from `npm install` surfaces naturally in the UI output. No special handling needed.

## 4. Fleet Config UI Additions

### New fields in `FleetConfigPanel`

**OPENCLAW_IMAGE** — plain text input, added to `fieldLabels`:
```ts
['OPENCLAW_IMAGE', t('openclawImage')],
```

**API Key** — password input, separate from `fieldLabels`. Only included in the `save(form)` payload when non-empty, so the masked display value is never echoed back:
```ts
// Separate state, not part of form
const [apiKey, setApiKey] = useState('');

// On save: only include if user typed something
const payload = { ...form, ...(apiKey ? { API_KEY: apiKey } : {}) };
```

**Enable npm packages** — checkbox toggle. Saved as `ENABLE_NPM_PACKAGES=true/false`. Displays a note: "Requires docker compose up -d to apply the mount change."

## Data Flow (Docker Mode, after this change)

```
First-time setup:
  1. User sets BASE_URL, API_KEY, MODEL_ID, OPENCLAW_IMAGE via Fleet Config UI
  2. User sets instance count and clicks Scale
  3. ComposeGenerator writes docker-compose.yml + per-instance openclaw.json
  4. docker compose up -d starts containers with full config

Plugin install:
  1. User opens Plugins tab on any instance
  2. Fleet manager calls POST /api/fleet/:id/plugins/install
  3. Server runs: docker exec openclaw-N node dist/index.js plugins install <spec>
  4. Plugin files → /home/node/.openclaw/extensions/ (config bind mount, persists)
  5. Config updated → /home/node/.openclaw/openclaw.json (config bind mount, persists)
  6. User restarts instance via Overview tab to load the plugin
```
